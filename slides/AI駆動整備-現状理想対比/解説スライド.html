GAP ANALYSIS — テスト系

自動検証基盤整備の現状と理想

AI がコード変更後に「壊れていない」と機械的に証明できる状態を目指す

■自動テストがなければ、AI の変更が正しいかどうかを人が毎回目視で確認するしかない

■テストコードを書くには、前提となるコンテキスト文書（観点表・設計書）の整備が先

テスト系 5 項目の現状・理想・ギャップ（上流から下流の依存順）
カテゴリ 現状 理想 ギャップ
1 テスト
フレームワーク		 未導入 フレームワーク未選定で自動テストが存在しない フレームワーク設定 言語・既存構成に応じたフレームワークを選定・導入済み 言語とビルド構成に合うフレームワークを選定し、導入する
2 単体テスト 未整備 関数の入出力仕様が未定義で、テストを書く基準がない テストコード 前提: 観点表（関数の引数・戻り値・例外条件の一覧） 観点表を整理し、単体テストコードを実装する
3 結合テスト 未整備 モジュール間の接続仕様が暗黙知のままになっている テストコード 前提: API 設計書 / データフロー定義 API 設計書を整備し、結合テストコードを実装する
4 E2E テスト 未整備 画面操作の期待動作が口頭伝達で、再現性がない テストコード 前提: 画面設計書（操作手順・画面遷移の期待動作） 画面設計書を整備し、E2E テストコードを実装する
5 カバレッジ
計測		 未導入 テストの網羅率を把握する手段がない CI 設定 目標値・除外対象を定義し、CI で自動計測 目標値を決め、CI にカバレッジ計測を組み込む

出典：対象プロジェクトの実地調査に基づく（調査時点の構成）

GAP ANALYSIS — 静的検査系

自動検証基盤整備の現状と理想

規約がなければ linter は意味を持たない — 言語化が先、自動化が後

■コーディングルールを言語化し linter に落とす — 規約が先、ツール導入が後

■「言語化 → 自動検証への変換」がこの基盤整備の本質

静的検査系 4 項目の現状・理想・ギャップ
カテゴリ 現状 理想 ギャップ
1 コード
linter		 未導入 規約が暗黙知で、linter 設定の根拠がない linter 設定 前提: コーディング規約（rules/ に命名規則・禁止パターンを定義） 規約を rules/ に文書化し、linter 設定に落とし込む
2 文体 linter 未導入 用語の表記揺れが放置され、読み手に負担がかかっている textlint 設定 前提: 文体規約（文長・読点数・冗長表現等）+ 用語辞書（prh.yml） 文体規約と用語辞書を整備し、textlint を導入する
3 リンク切れ
検査		 未導入 リンク切れの検知が目視頼みで見落としが発生する linkcheck 設定 前提: 検査対象範囲と除外パターンの定義 検査範囲を定義し、リンク検査を導入する
4 HTML
静的検査		 未導入 HTML 品質検査の基準が未定義で、検査が実施できない linter 設定 前提: 検査観点（rules/ に必須属性・アクセシビリティ等を定義） 検査観点を rules/ に定義し、linter 設定に変換する

linter の設定は規約文書から導出する — 規約がなければ linter は意味を持たない

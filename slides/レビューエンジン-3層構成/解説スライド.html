レビューエンジン 3層構成

機械的チェック層 — 人が見る前に機械が潰す

全対象で共通の3種の自動検査が、語彙・文体・略語の指摘をコミット前に消す

1

置き換え辞書による語彙チェック

カタカナ語・英語バズワードを平易な日本語へ自動置換する辞書。登録語を検出し、推奨語へ置き換える

アジェンダ→議題 リソース→資源 ナレッジ→知見 レビュー→照合
語彙の置き換え後に、文の構造を検査する
2

textlint 文書品質チェック

文の長さ・読点の数・漢字連続など、読みやすさの定量基準を機械検査する

1文100字以内 読点3個以内 漢字連続7文字以内 冗長表現の簡潔化
文体の基準通過後に、略語の定義を検査する
3

略称照合チェック

説明なしの略称・頭字語をブロックリストで検出し、初出時に正式名称の定義を強制する

ref→参照 SHA→ハッシュ値 HMR→Hot Module Replacement
レビューエンジン 3層構成

判断系レビュー層 — 4つの専門家が観点別に照合する

4 専門家は常に全員が動くわけではない。何を提示する前のレビューかで異なる組み合わせが選ばれる

文書照合

文書成果物を規約に照合する。形式・構成・文節改行・意味キーの 4 観点

  • 被覆率・図解候補・禁止パターンの照合
  • 自己完結性・外部リソース非依存の検査
  • 日本語の文節境界での改行品質
  • 連番禁止・意味語キーの検査

コード照合

コード差分を規約に照合する。共通基準と UI 実装基準の 2 観点

  • OWASP 脆弱性・エラー処理・命名規約の照合
  • アイコン統一（Material Symbols 必須）・ホバー非依存

反証

作成文脈を持たない独立した懐疑者として反証を試みる。「誤りがある」前提で証拠を探す

  • 論理整合・範囲整合・影響範囲の 3 観点で反証
  • コード差分に対して反証（変更が壊すものを探す）
  • 合否の宣言はしない。証拠付きの反証所見を返すのみ

初見読解

弱モデルの初見読解で計画の暗黙知を炙り出す

  • 弱モデルが計画どおりに実行できるか検証
  • 初見の読み手が成果物を理解できるか検証
  • 理解できなかった部分 = 暗黙知が省略されている証拠

レビューエンジン 3層構成

可変レビュー層 — 対象から規約を動的に解決する

計画文の触る領域や変更ファイルのパスから、適用すべき規約を自動で特定する

計画の領域から規約を特定

計画文の触る領域から適用規約を特定する

  • 計画文が UI を触る → UI 実装基準を照合
  • 計画文が DB を触る → DB 設計基準を照合
  • 計画文が API を触る → API 設計基準を照合
  • 計画文が認証に触る → セキュリティ基準を照合
  • 計画文がカタログに触る → カタログサイト基準を照合

計画の提示前に発火

変更ファイルから規約を特定

変更ファイルのパスから適用規約を特定する

  • *.html の変更 → HTML 出力基準を照合
  • src/components/ → UI 実装基準を照合
  • *.test.* の変更 → テスト基準を照合
  • *.sh の変更 → シェルスクリプト基準を照合
  • schema.prisma の変更 → DB 設計基準を照合

実装の完了後に発火

成果物の提示前では可変レビューを実行しない（共通基準のみで照合する）

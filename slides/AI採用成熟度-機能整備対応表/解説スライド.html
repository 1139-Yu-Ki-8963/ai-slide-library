AI 採用 成熟度モデル

AI 採用の 5 段階 — 各段階の状態と導入すべき機能

段階を飛ばさない — ガードレールを整備してから機能を解放する

1 導入基盤を作る
2 並列で動かす
3 自律に委ねる
LEVEL 1

利用制限
Gated

AI ツールの利用が厳しく制限されている。古いモデル・高遅延・承認未整備

壁: レガシー承認プロセス。非技術者が意思決定を支配

使う機能

—（AI 未導入）

必要な整備
  • SSO/SCIM + 役割ベースアクセス制御
  • 組織レベルの予算上限
  • 既存承認フローへの組込み

→ 経営層と合意し導入枠組みを構築

LEVEL 2

1対1ペア作業
Assisted

1人+1エージェントの同期作業。全出力を手動確認。信頼不足で書き直しが多い

壁: 出力を全件読まないと不安。同期型で遅い

使う機能
  • Claude Code（Desktop/CLI/IDE）
  • Claude Cowork / Design
  • Analytics ダッシュボード
  • Compliance API
必要な整備
  • 1人あたりの利用上限
  • 中央管理のモデル設定
  • OpenTelemetry で既存監視に連携

→ 複数エージェント運用と自己検証ループの確立

LEVEL 3

並列オーケストレーター
Parallel

1人が5〜10エージェントを同時運用。各自が worktree でテスト・ビルド・スキャンまで自走

壁: 手動レビューが律速。出力量に追いつかない

使う機能
  • Auto mode
  • Agent view
  • Claude Code Review
  • Claude Security Review
  • Worktree isolation
  • Remote control（スマホ監視）
必要な整備
  • 自動コードレビュー・セキュリティレビュー
  • E2E 検証パイプライン
  • 事前承認された安全なコマンド
LEVEL 4

監督付き自律
Supervised

AI がコードの大半を書く。質問が「コード読んだ？」→「次は何をする？」に変化。メンテがバックグラウンド化

壁: ループへの信頼。トークン使用量の管理

使う機能
  • Subagents + Worktree isolation
  • Dynamic workflows
  • Routines / loop / batch
  • Claude Tag（使用状況監視）
必要な整備
  • 自動コードレビュー・セキュリティレビュー
  • Agent sandbox
  • CLAUDE.md の整備
  • モデル選択によるトークン制御
LEVEL 5

AI ネイティブ
AI-native

ループ完全閉鎖。エージェントがエージェントを管理・起動・停止。人間は「意図」で指示し例外だけ監視

壁: ドメイン特化の自動化を安全にスケール

使う機能
  • Claude Agent SDK
  • Claude Tag（Slack 自動応答）
必要な整備
  • 自動化向けのコスト制御
  • 自動化向けのモデル選択
!

よくある失敗: ガードレールなしに Level 3 の並列運用に飛びつく → 自動レビューがないまま出力が溢れ、品質が崩壊する

1

導入基盤を作る（Level 1〜2）— 承認の壁を取り、1対1ペアで信頼を築く

2

並列で動かす（Level 3）— Auto mode と自動レビューで 5〜10 エージェントを同時運用

3

自律に委ねる（Level 4〜5）— AI 主導で書き、人間は意図と例外に集中
